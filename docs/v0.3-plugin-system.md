# Dofek v0.3 — Plugin System Plan

## Context

Dofek v0.2 ships a trading-terminal TUI with a 3-line "plugin dock" placeholder in the watchlist panel (`src/ui/watchlist.rs:241-255`). v0.3 turns this into a real plugin system that lets external processes inject custom data panels, process annotations, and metrics into the dashboard.

**Design decisions (locked in):**
- JSON-over-stdio protocol (language-agnostic, simple)
- Pull model (Dofek polls plugins on each refresh cycle)
- Dofek spawns and manages plugin child processes

---

## 1. Plugin Protocol

### 1.1 Communication Flow

```
dofek                          plugin (child process)
  |                                |
  |-- spawn plugin process ------->|
  |                                |
  |  [every refresh_ms tick]       |
  |-- {"type":"poll"} + \n ------->|
  |<-- JSON response + \n ---------|
  |                                |
  |  [on shutdown]                 |
  |-- {"type":"shutdown"} + \n --->|
  |-- kill process after 2s ------>|
```

- Messages are newline-delimited JSON (one JSON object per line on stdio)
- dofek writes to plugin's **stdin**, reads from plugin's **stdout**
- Plugin's **stderr** is captured for logging/diagnostics

### 1.2 Poll Request (dofek -> plugin)

```json
{
  "type": "poll",
  "timestamp_ms": 1713020400000,
  "processes": [
    { "pid": 1234, "name": "ollama_llama_server.exe", "vram_bytes": 4294967296 }
  ]
}
```

The `processes` field gives plugins context about currently running processes so they can enrich/annotate them. Plugins that don't care about processes ignore it.

### 1.3 Poll Response (plugin -> dofek)

```json
{
  "type": "poll_response",
  "status": "ok",
  "panels": [
    {
      "id": "ollama-models",
      "label": "OLLAMA",
      "content": [
        { "key": "Model", "value": "llama3:8b", "style": "accent" },
        { "key": "Status", "value": "idle", "style": "dim" }
      ]
    }
  ],
  "process_annotations": [
    {
      "pid": 1234,
      "label": "llama3:8b",
      "category": "ai",
      "ai_state": "idle"
    }
  ],
  "metrics": [
    {
      "id": "ollama.requests_per_min",
      "label": "Req/min",
      "value": 12.5,
      "unit": "req/m"
    }
  ]
}
```

All three top-level arrays (`panels`, `process_annotations`, `metrics`) are optional. A plugin can provide any combination.

| Field | Purpose |
|---|---|
| `panels` | Key-value data rendered in the plugin dock |
| `process_annotations` | Enrich process rows with labels, override category/ai_state |
| `metrics` | Named numeric values shown in the ticker bar or available for charting |

### 1.4 Handshake (first poll response)

On the first `poll`, the plugin should include a `manifest` field:

```json
{
  "type": "poll_response",
  "manifest": {
    "name": "dofek-ollama",
    "version": "0.1.0",
    "description": "Ollama model status and inference tracking",
    "author": "dofek contributors"
  },
  "status": "ok",
  "panels": []
}
```

The manifest is only read once. Subsequent responses omit it.

### 1.5 Style Values

Panel content `style` field accepts: `"normal"`, `"accent"`, `"dim"`, `"warn"`, `"error"`. Mapped to theme colors by dofek.

## 2. Plugin Lifecycle

### 2.1 Configuration

```toml
# dofek.toml

[[plugins]]
name = "ollama"
command = "dofek-ollama"          # resolved via PATH or absolute path
args = ["--port", "11434"]        # optional
enabled = true                    # default true
timeout_ms = 2000                 # per-poll timeout, default 2000
```

Multiple `[[plugins]]` entries for multiple plugins. Order determines dock layout order.

### 2.2 Spawn & Supervision

```
startup:
  for each enabled plugin in config:
    spawn child process (stdin/stdout piped, stderr captured)
    attach to Windows Job Object (auto-kill on parent exit)
    mark as "starting"

refresh loop:
  for each plugin:
    if process dead:
      increment crash_count
      if backoff_elapsed (1s, 2s, 4s, 8s, 16s, 30s cap):
        respawn
      else:
        skip
    else:
      write poll request to stdin
      read response with timeout (timeout_ms)
      if timeout or parse error:
        increment error_count
        if error_count > 5 consecutive:
          log warning, mark as "unhealthy"
          (keep polling — don't kill)
      else:
        reset error_count
        merge response into AppData

shutdown:
  send {"type":"shutdown"} to each plugin
  wait 2s
  kill remaining
```

### 2.3 Plugin States

```
Starting -> Healthy -> Unhealthy (5+ consecutive errors)
    |          |            |
    v          v            v
  Crashed   Crashed      Crashed
    |          |            |
    v          v            v
  Restarting (backoff)  Restarting (backoff)
```

State is displayed in the plugin dock UI next to the plugin name.

## 3. Implementation Plan

### 3.1 New Modules

| Module | Purpose |
|---|---|
| `src/plugin/mod.rs` | `PluginManager`: spawn, poll, restart, shutdown |
| `src/plugin/protocol.rs` | Serde structs for request/response JSON |
| `src/plugin/process.rs` | Child process wrapper: stdio pipes, timeout reads, Windows Job Object |

### 3.2 Changes to Existing Modules

| File | Change |
|---|---|
| `src/config.rs` | Add `plugins: Vec<PluginConfig>` to `Config` struct |
| `src/app.rs` | Add `plugin_data: HashMap<String, PluginResponse>` to `App`. Merge annotations into process list |
| `src/data/mod.rs` | Expose minimal process list for plugin poll context |
| `src/data/process.rs` | Add optional `plugin_label: Option<String>` to `ProcessInfo` |
| `src/ui/watchlist.rs` | Replace placeholder with real plugin dock rendering from `plugin_data` |
| `src/ui/ticker.rs` | Render plugin-provided metrics as additional ticker pills |
| `src/main.rs` | Init `PluginManager`, integrate into refresh loop |

### 3.3 Threading Model

Plugins run as child processes (separate from dofek threads). The `PluginManager` lives on the **data collector thread** alongside sysinfo/NVML polling:

```
Data collector thread:
  loop {
    refresh sysinfo
    query NVML
    query network
    poll_all_plugins()      // <-- new: write stdin, read stdout for each plugin
    send DataSnapshot       // now includes plugin_data
  }
```

No new threads needed. Plugin I/O is synchronous with timeouts. If a plugin is slow, it hits the timeout and gets skipped for that tick — other plugins and core data collection are not blocked because each plugin poll has an independent timeout.

### 3.4 Build Order

**Phase 1 — Protocol & plumbing (no UI changes)**
1. Define serde structs in `src/plugin/protocol.rs`
2. Implement child process wrapper with stdio pipes + timeout in `src/plugin/process.rs`
3. Implement `PluginManager` (spawn, poll loop, restart logic) in `src/plugin/mod.rs`
4. Add `PluginConfig` to `src/config.rs`
5. Wire `PluginManager` into data collector thread
6. Add `plugin_data` to `App` state

**Phase 2 — UI integration**
7. Replace plugin dock placeholder with real rendering
8. Show plugin status (healthy/unhealthy/crashed) with colored indicator
9. Render plugin panels as key-value rows in the dock
10. Apply `process_annotations` to process table rows
11. Render plugin `metrics` in ticker bar

**Phase 3 — Reference plugins**
12. `dofek-ollama` — queries Ollama API, reports loaded models, inference state, annotates Ollama processes
13. `dofek-docker` — lists running containers, resource usage, annotates container processes

## 4. Reference Plugin: dofek-ollama

Standalone binary (Rust or Python). On each poll:
1. Calls `GET http://localhost:11434/api/tags` for loaded models
2. Calls `GET http://localhost:11434/api/ps` for running models
3. Matches PIDs from `processes` array to Ollama workers
4. Returns panel with model names/status + process annotations

```toml
[[plugins]]
name = "ollama"
command = "dofek-ollama"
args = ["--host", "http://localhost:11434"]
```

---

## 5. Reference Plugin: dofek-docker

Standalone binary. On each poll:
1. Calls Docker Engine API (`/containers/json`) via named pipe or TCP
2. Reports running containers with CPU/MEM stats
3. Annotates container processes in the process table

```toml
[[plugins]]
name = "docker"
command = "dofek-docker"
```

## 6. Plugin Dock UI

### Current (v0.2)
```
┌─ PLUGINS ──────────────────────┐
│ No plugins connected           │
└────────────────────────────────┘
```

### With plugins (v0.3)
```
┌─ PLUGINS ──────────────────────┐
│ ● OLLAMA  llama3:8b idle       │
│ ● DOCKER  3 containers         │
└────────────────────────────────┘
```

- Green dot = healthy, yellow = unhealthy, red = crashed
- Dock height scales: 3 lines base + 1 per additional plugin panel row
- If dock grows too tall, cap at 6 lines and allow scrolling (Down/Up when dock focused)

## 7. Verification

### Manual testing
1. Build dofek with plugin support, no plugins configured — dock shows "No plugins connected" (no regression)
2. Add a `[[plugins]]` entry pointing to a simple echo plugin (responds with static JSON) — verify dock renders panel
3. Kill the echo plugin process — verify crash detection, backoff restart, status indicator turns red then green
4. Configure `timeout_ms = 100` with a slow plugin — verify timeout handling, unhealthy state after 5 failures
5. Test `dofek-ollama` with a running Ollama instance — verify model names appear in dock, process annotations show in watchlist
6. Exit dofek — verify all child processes are cleaned up (no orphans in Task Manager)

### Automated tests
- Unit tests for protocol serde (round-trip serialize/deserialize)
- Unit tests for backoff timing logic
- Integration test: spawn a mock plugin script, verify poll/response cycle

## 8. Future Extensions (out of scope for v0.3)

- Plugin-provided chart series (plugins emit time-series data plottable in the main chart area)
- Plugin commands (dofek sends user-triggered actions to plugins, e.g., "unload model")
- Plugin discovery (scan PATH for `dofek-*` binaries, auto-suggest in config)
- Remote plugins (TCP/WebSocket instead of stdio — for cross-machine monitoring)
- Plugin SDK crate (`dofek-plugin-sdk`) with derive macros for response structs
