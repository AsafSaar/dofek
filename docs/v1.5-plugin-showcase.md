# Plan: Turn `dofek-ollama` + `dofek-docker` into Showcase Plugins

> Note: per user preference, this plan should also be saved as `docs/v1.5-plugin-showcase.md` once approved. Plan-mode constraints prevent that edit now.

## Context

dofek ships two first-party plugins under `plugins/`: `dofek-ollama` and `dofek-docker`. They're billed on the website as "official plugins" and meant to inspire third-party plugin authors. Reviewing them shows:

- **Both are functional, not demos.** Each is ~270 LOC of Rust, queries a real local API (Docker Engine `/containers/json`, Ollama `/api/ps` + `/api/tags`), gracefully degrades when the service is offline, and faithfully implements every feature of the JSON-over-stdio protocol: manifest on first poll, panels, process annotations (with `category` + `ai_state`), metrics, and shutdown.
- **But they're minimum-viable showcases.** Each plugin emits exactly 1 panel and 1 metric. No per-plugin README. They duplicate the protocol structs locally instead of importing a shared crate. They are **not built by `build-all.sh`/`build-all.ps1`** and **not bundled in MSI/deb/rpm installers** — `dofek.toml.example` only references them as commented-out examples (lines 43–53). The website lists them but provides no install instructions or downloads.

The goal of this work: make these two plugins the kind of polished, easy-to-try, easy-to-copy reference that turns curious users into plugin authors.

## Status of each plugin

| Aspect | dofek-ollama | dofek-docker |
|---|---|---|
| Lines of code | 301 (`plugins/dofek-ollama/src/main.rs`) | 258 (`plugins/dofek-docker/src/main.rs`) |
| Backend | Ollama HTTP API (`localhost:11434`) | Docker Engine API (`localhost:2375`) |
| Panels emitted | 1 (`ollama-status`) — running models + sizes, available count | 1 (`docker-status`) — up to 3 containers + image counts |
| Metrics emitted | 1 (`ollama.running`) | 1 (`docker.containers`) |
| Process annotations | ollama → `category=ai`, model label, `ai_state=idle` | docker/containerd → `category=dev` |
| Offline behaviour | Returns "offline" entry, status still ok | Returns "offline" entry, status still ok |
| README | None | None |
| Built by `build-all.*` | No | No |
| Listed on website | Yes (cards + playground example) | Yes (cards + playground example) |
| Code health | Solid; 5 unwraps in safe HTTP/serde paths; protocol structs duplicated locally | Same shape; same duplication |

Both use `edition = "2024"` (correct — matches the workspace; the rest of dofek is already on it).

## Recommended approach

Treat the showcase work as four small, independent tracks. None of them require breaking changes; all of them strengthen the "obviously copy-paste-able" feel.

### Track 1 — Per-plugin README (highest leverage, smallest change)

Add `plugins/dofek-ollama/README.md` and `plugins/dofek-docker/README.md` covering:

- One-paragraph "what it shows you in dofek" with a screenshot of the panel/dock
- Prerequisites (Ollama running on 11434, or Docker daemon exposed on 2375)
- Build: `cargo build --release -p dofek-<name>`
- Install: where to drop the binary so dofek finds it on PATH
- `dofek.toml` snippet (copy-paste)
- Troubleshooting (most common: Docker socket vs TCP, Ollama not running)
- Link back to the protocol guide at `plugins/README.md`

This single change does more for "showcase" perception than anything else.

### Track 2 — Extract `dofek-plugin-protocol` crate

Today both plugins re-declare the request/response structs (`plugins/dofek-ollama/src/main.rs:173–301`, `plugins/dofek-docker/src/main.rs:189–258`). The canonical types already exist in `src/plugin/protocol.rs`.

Move that file into a new workspace crate `crates/dofek-plugin-protocol/` (or `plugin-protocol/`) and:

- Have the dofek main binary depend on it via path
- Have both plugins depend on it via path (same workspace) and via version once published
- Eventually publish to crates.io so external authors can `cargo add dofek-plugin-protocol`

This turns the Rust example in `plugins/README.md` (lines 249–307) from "here's a 50-line skeleton you copy" into "add one dep, write your `poll()` function". The Python example stays unchanged.

### Track 3 — Build & ship the plugins with releases

`build-all.ps1` and `build-all.sh` currently build only `dofek-tui` + `dofek-gui` (lines 18–29 of each). Extend them to also build the two plugins and stage their binaries alongside the main artefacts:

- Windows: include `dofek-ollama.exe` + `dofek-docker.exe` next to the MSI as standalone downloads (don't bundle into the MSI yet — they're optional)
- Linux: ship as separate `.deb`/`.rpm` packages (`dofek-plugin-ollama_1.5.0_amd64.deb`, etc.) that drop the binary in `/usr/bin`
- macOS: ship as separate signed binaries

Update the GitHub release notes template to mention plugins are optional add-ons. Update `dofek.toml.example` lines 43–53 to flip from commented-out to a clear "uncomment to enable if installed" block.

### Track 4 — Add one more first-party plugin to show *variety*

The two existing plugins are very similar (HTTP poll → panel + metric). A third plugin that exercises a *different* shape would teach more:

Pick one of:
- **`dofek-net-ping`** — pings a configurable host, emits latency metric + state. Demonstrates `args` config, stateful sample buffer, periodic work between polls.
- **`dofek-spotify`** — reads currently-playing track via the local Spotify Web Helper. Demonstrates rich panel content + auth/credential handling via env (links to global CLAUDE.md `.env` rule).
- **`dofek-uptime-kuma`** — hits a remote Uptime Kuma status page. Demonstrates non-localhost dependencies and longer poll cadence.

Recommendation: `dofek-net-ping` — it's the most universal, has zero auth surface, and the latency time-series naturally fills the "metric" pill in a way the existing two don't.

- `src/plugin/protocol.rs` is already the source of truth for the JSON schema — Track 2 just relocates it.
- `src/plugin/process.rs` already handles spawn/timeout/Job-Object lifecycle for plugins — no changes needed.
- The website's `playground.js` already renders both plugins as live JSON examples; once Track 1 lands, link from the cards into the new READMEs.

## Verification

1. **Per-plugin docs** — `cargo build --release -p dofek-ollama && ./target/release/dofek-ollama < <(echo '{"type":"poll","timestamp_ms":0,"processes":[]}')` produces a valid response; the README's copy-paste `dofek.toml` snippet boots cleanly under `cargo tui`.
2. **Protocol crate** — `cargo build` from workspace root succeeds; `cargo tree -p dofek | grep dofek-plugin-protocol` shows the dependency; both plugins still emit byte-for-byte identical JSON to the pre-refactor version (capture via `cargo run -p dofek-ollama | head -2` before/after).
3. **Release pipeline** — run `./build-all.sh` (or `.ps1`); confirm `target/release/dofek-{ollama,docker}` (or `.exe`) exist and are stripped/release-mode; confirm the `.deb`/`.rpm`/MSI staging directory contains them as separate artefacts.
4. **End-to-end** — start Ollama, run `cargo tui` with both plugins enabled in `dofek.toml`, confirm the dock shows both panels with green health dots, the ticker bar shows `ollama.running` + `docker.containers` pills, and the ollama process row in the watchlist shows the model label.
5. **Website** — load `website/plugins/index.html` locally; confirm the playground still parses both example responses; confirm new install links resolve.

## Out of scope (deliberately)

- Bundling plugins *inside* the main MSI/deb/rpm — keep them as optional add-ons.
- Async runtime in plugins (tokio). Sync `ureq` is fine and matches dofek's overall sync architecture.
- AMD GPU / non-NVIDIA telemetry plugins (separate roadmap item, blocked on driver coverage).
- Auto-update for plugins. Out of scope for v1.5; revisit if a plugin marketplace ever materialises.
