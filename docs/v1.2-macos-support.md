# macOS (Apple Silicon) support for dofek

## Context

dofek is a Rust dual-interface system monitor (Ratatui+crossterm TUI; Tauri 2 GUI) currently shipping for Windows x64 and Linux x86_64 (v1.1.0, April 2026). The repo's own CLAUDE.md flags macOS and ARM64 as deferred non-goals. This plan adds first-class macOS support so Apple Silicon users can run both the TUI and the GUI from a normal release artifact, closing the most-requested platform gap and removing the largest item from the "future releases" list. Architecturally, dofek is already mostly portable — TUI rendering, sysinfo-based collection, config paths, the Unix `kill(SIGTERM)` path, and the plugin protocol are platform-neutral or already covered by `cfg(unix)`. The work concentrates in three small code touchpoints, two CI files, and three docs files.

## Scope (locked with user)

- **Architecture:** Apple Silicon only — `aarch64-apple-darwin`. No Intel, no universal binary.
- **GPU/VRAM/CPU-temp on macOS:** deferred. Panels show N/A. Documented as known limitation.
- **Packaging:** unsigned `.app` bundle (zipped) + standalone TUI binary. No signing, no notarization, no DMG, no Homebrew.
- **CI:** lint+test job and release-build job added in the same PR.
- **Version:** bump to `1.2.0` (minor — new platform, no breaking changes).

## Code changes

### 1. `src/lib.rs` — add `cfg(target_os = "macos")` arm to `os_version_string()`
Currently three arms (lib.rs:16, lib.rs:35, lib.rs:49). The fallback at lib.rs:49 returns just `"macos"`. Add a fourth explicit arm and narrow the fallback to `not(any(windows, target_os = "linux", target_os = "macos"))`.

Body invokes `sw_vers -productName` and `sw_vers -productVersion` via `std::process::Command`, then maps the major version to a codename (15→Sequoia, 14→Sonoma, 13→Ventura, 12→Monterey, 11→Big Sur). Output examples: `macOS 14.6 (Sonoma)`, or `macOS 16.0` for unknown future versions. No new crate dependency.

### 2. `src/data/network.rs` — extend the loopback filter
The `cfg(not(windows))` arm at network.rs:108 already handles macOS via `sysinfo::Networks`. Replace the single-line filter at network.rs:126 (`if name == "lo" { continue; }`) with:
- `lo` and `lo0` skipped on all Unix.
- An additional `cfg(target_os = "macos")` block that skips Apple-internal pseudo-interfaces: `gif0`, `stf0`, `awdl0`, `llw0`, `anpi0`, `anpi1`, `ap1`. These never carry user-visible traffic.
- Keep `utun*` and other VPN/container interfaces visible — users want to see them.

### 3. Files that already work — no code change required
- `src/app.rs:849-869` (TUI process kill) — `cfg(unix)` SIGTERM path covers macOS.
- `gui/src/lib.rs:125-144` (Tauri kill command) — same.
- `src/plugin/process.rs` — Windows-only `CREATE_NO_WINDOW` is already correctly gated.
- `src/data/lhm.rs` — fails fast on non-Windows; existing graceful-degradation path.
- `src/data/gpu.rs` — `nvml_wrapper::Nvml::init()` returns `Err` when libnvidia-ml is absent; already handled. **Risk to verify on real Mac:** that init does not panic during `libloading` on darwin (mitigation: wrap in `catch_unwind` if it does).
- `src/data/sysinfo_source.rs:70` and `src/data/mod.rs:72-92` (CPU temperature) — leave as `cfg(target_os = "linux")`. macOS thermal sensor coverage in `sysinfo` is insufficient on Apple Silicon; returning `None` is correct and the UI already renders `--`.
- `Cargo.toml` — `[target.'cfg(unix)'.dependencies]` already pulls `nix` for macOS. Only metadata change: update `description` on line 8 to mention macOS.

### 4. Tauri config — no functional change
`gui/tauri.conf.json` `externalBin: "../target/release/dofek-tui"` — Tauri suffixes with the host triple, so on macOS-arm64 it expects `dofek-tui-aarch64-apple-darwin`. The existing `build-all.sh` already produces this. No config change needed.

### 5. Version bump
- `Cargo.toml`: `version = "1.1.0"` → `"1.2.0"`
- `gui/tauri.conf.json`: `"version": "1.1.0"` → `"1.2.0"`

## CI changes

### `.github/workflows/ci.yml` — add `check-macos` job
Append a third job mirroring `check-linux` but on `macos-latest` (which is now arm64 by default), with toolchain target `aarch64-apple-darwin`. No system-package install step needed — Xcode CLT is preinstalled. Steps: checkout → toolchain → rust-cache → `cargo clippy --all-targets -- -D warnings` → `cargo test`.

### `.github/workflows/release.yml` — add `build-macos` job
Insert between `build-linux` (release.yml:109) and `release` (release.yml:204). Mirrors the Linux job structure with these macOS-specific changes:
- `runs-on: macos-latest`, target `aarch64-apple-darwin`
- No system-package install step
- `cargo install tauri-cli --version '^2.0' --locked`
- `cargo tauri build --bundles app` (suppresses DMG generation; unsigned DMGs are useless to end users)
- Artifact collection: `dofek-tui` standalone, plus `find target/release/bundle/macos -name '*.app' -type d` zipped to `dofek-gui-aarch64-apple-darwin.app.zip` (`.app` is a directory, must be zipped to upload)
- `shasum -a 256` (BSD variant) instead of `sha256sum`
- Upload as artifact name `macos-build`

Update the `release` job at release.yml:204:
1. `needs: [build-windows, build-linux]` → `[build-windows, build-linux, build-macos]`
2. Add `Download macOS artifacts` step (path `artifacts-macos`)
3. Update the `Stage artifacts and combine checksums` step to include `artifacts-macos` in the `find` and `cat`
4. Add a "macOS (Apple Silicon — aarch64)" section to the release-body markdown listing the two artifacts
5. Update the Gatekeeper warning line: "On macOS, Gatekeeper will block first launch — right-click the .app → Open → Open, or run `xattr -dr com.apple.quarantine dofek.app`."

## Verification (run on an Apple Silicon Mac)

**Pre-flight**
1. `xcode-select -p` returns a path; `cargo --version` works; `rustup target list --installed` includes `aarch64-apple-darwin`.
2. `cargo install tauri-cli --version '^2.0' --locked` (one-time).

**TUI**
3. `cargo build --release -p dofek --bin dofek-tui` builds clean with no platform warnings.
4. `file target/release/dofek-tui` → `Mach-O 64-bit executable arm64`.
5. `./target/release/dofek-tui` launches and renders. Confirm:
   - CPU per-core load animates; MEM panel shows used/total
   - NET panel shows real interfaces; no `lo0`, no `awdl0`/`llw0`/`gif0`/`stf0`
   - GPU panel renders (no crash) with `--` / N/A
   - CPU temp shows `--` (documented limitation)
   - Header OS string reads e.g. `macOS 14.6 (Sonoma)`, not `macos`
   - Process list populates; pressing `x` on a safe test process (e.g. a `cat` pipe) kills it via SIGTERM
   - `q` quits cleanly

**GUI**
6. `./build-all.sh` produces `target/release/bundle/macos/dofek.app/`.
7. `xattr -dr com.apple.quarantine target/release/bundle/macos/dofek.app && open target/release/bundle/macos/dofek.app` — window opens, all panels match TUI behavior, kill-process action works.
8. After quitting, `ps aux | grep dofek` shows no orphaned `dofek-tui`.

**CI**
9. Push branch — `check-macos` is green; `check-windows` and `check-linux` still pass.
10. Push tag `v1.2.0-rc1` — `build-macos` produces `dofek-tui` and `dofek-gui-aarch64-apple-darwin.app.zip`; combined `SHA256SUMS.txt` includes both; draft release lists all three platforms.

**Released-artifact smoke test**
11. Download the zipped `.app` from the draft release on a fresh user account, unzip, drag to `/Applications`, right-click → Open → Open. Gatekeeper prompts once, then the app runs.
12. Download `dofek-tui` standalone; `xattr -d com.apple.quarantine ./dofek-tui && chmod +x ./dofek-tui && ./dofek-tui` — runs.

## Risks and known unknowns

- **NVML graceful failure on darwin** — verified in CI step 9 (runners have no NVIDIA hardware; if init panics, the lint/test step fails). Mitigation if it panics: wrap `Nvml::init()` in `std::panic::catch_unwind` in `src/data/gpu.rs`.
- **`cargo tauri build --bundles app` flag syntax** — Tauri 2 CLI flags have shifted across releases; if `--bundles app` is rejected, fallback is to add a `bundle.targets: ["app"]` macOS override in `tauri.conf.json` (this is the safer long-term option but requires touching the JSON config, so we try the CLI flag first).
- **`sw_vers` under a Tauri sandbox** — Tauri 2 doesn't apply hardened runtime by default to unsigned apps; subprocess should work. Verified by the GUI step displaying the proper OS string.
- **macOS pseudo-interface name list** — Apple occasionally adds new ones (`bridge100` for hotspot, etc.). The filter is intentionally conservative; new synthetics will appear in the panel until the filter is extended. Acceptable.
