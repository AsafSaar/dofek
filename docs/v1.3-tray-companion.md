# dofek v1.3 — tray companion + RAPL CPU power + disk I/O

## Context

dofek is a Rust dual-interface system monitor (Ratatui+crossterm TUI; Tauri 2 GUI) currently shipping for Windows, Linux, and macOS Apple Silicon (v1.2.0 just shipped — RC1 build run is in progress). The system-tray companion has been on the roadmap since day one and is the user's "leading feature" — they want a live-rendering icon in the tray/menu-bar so the OS chrome itself becomes a system monitor. Two adjacent quick wins go in the same release: Linux CPU power via RAPL (closes a long-standing N/A on Linux) and disk I/O metrics (the most-requested missing dimension after GPU). Architecturally, dofek's data layer already produces a single `DataSnapshot` consumed by both UIs, so the tray reuses that stream — no second collector, no IPC duplication.

## Scope (locked with user)

- **Tray companion (headline)**: live-updated icon (re-rendered every ~500ms with a tiny CPU sparkline drawn into the tray PNG); on macOS the menu bar additionally shows a short text label `CPU NN GPU NN`. Right-click menu (Show / Hide / Settings… / Quit). Left-click toggles the window. All three platforms. **Close-to-tray default**: pressing the window close button hides to tray; quit only via tray menu / Cmd+Q / Alt+F4. Setting to disable.
- **Linux RAPL CPU power**: read `/sys/class/powercap/intel-rapl:0/energy_uj` deltas → watts → populate `cpu.power` on Linux. Graceful failure on permission-denied / non-Intel.
- **Disk I/O**: cross-platform via `sysinfo::Disks`. New `ChartTab::Disk` (TUI key `d`, GUI new tab). Aggregate read/write rates in a ticker pill (gated to appear only on real activity). No 5th bottom-strip panel in v1.3 — keep the existing 4-panel layout untouched.
- **Version**: bump 1.2.0 → 1.3.0. While bumping, replace the hardcoded `" v1.2"` literal at `src/ui/ticker.rs:22` with `concat!(" v", env!("CARGO_PKG_VERSION"))` so the TUI ticker self-syncs forever.

### B. Linux RAPL CPU power

#### `src/data/rapl.rs` (new)
`pub struct RaplTracker { prev_uj, prev_at, max_uj, disabled }`. `pub fn read_watts(&mut self) -> Option<f32>` reads `/sys/class/powercap/intel-rapl:0/energy_uj` (microjoule monotonic counter, may wrap), takes time delta, returns watts. Reads `max_energy_range_uj` once for wraparound handling. Sets `disabled = true` on first read failure to keep the hot loop quiet on hardened distros.

## Verification (per OS)

Run on macOS Apple Silicon, Windows 11, and a KDE-based Ubuntu (so we test outside GNOME's gated tray support):

1. **Tray appears on launch**, shows a flat baseline icon. Within 1-2s the icon visibly updates as CPU varies (induce with a stress loop).
2. **Left-click toggles window** (visible → hide → visible+focus).
3. **Right-click menu** shows Show / Hide / Settings… / Quit; Quit fully exits (`pgrep dofek` empty).
4. **macOS menu-bar text**: `CPU NN GPU NN` next to the icon when `tray_show_text=true`; toggle off in help overlay → text disappears, icon stays.
5. **Close-to-tray**: window close → window hidden, process alive, tray persists. Toggle `close_to_tray=false` → close button quits.
6. **Start-in-tray**: toggle on, restart → app launches with no visible window, tray only.
7. **Single-instance** (Win/Linux): launch dofek-gui twice → second invocation just focuses the existing window (no second tray).
8. **Linux RAPL** (Intel host): CPU pill in ticker shows watts; non-Intel or hardened-perm host shows nothing (no errors in `RUST_LOG=info`).
9. **Disk I/O — TUI**: press `d` → DISK tab with read/write area stack; ticker pill appears under heavy I/O (`dd if=/dev/zero of=/tmp/x bs=1M count=1024`).
10. **Disk I/O — GUI**: click DISK tab; same.
11. **Settings persistence**: toggle each new setting, kill GUI, relaunch → state survives in `settings.toml`.
12. **GNOME smoke test** (if available): without AppIndicator extension installed, document tray absence in release notes; with it installed, tray renders.

## Risks and known unknowns

- **`tiny-skia` on the GUI bundle** — adds ~150KB. Only on `dofek-gui`, never on the lean `dofek-tui`. Acceptable; alternative is hand-rolling a 32×32 RGBA buffer (≈80 lines of pixel arithmetic, no AA) — defer to v1.4 if anyone cares.
- **macOS title text width budget** — `CPU 100 GPU 100` is 12 chars and fits on every Mac sold in the last decade. If users want hostname or AI-count in the menu bar, surface a `tray_format` template setting in v1.4.
- **GNOME tray** — Shell removed legacy tray. Document the AppIndicator extension requirement in CHANGELOG and the manual; we are not shipping a workaround.
- **RAPL permission** — `disabled = true` on first failure means no spam; the cost is no power on hardened distros. Acceptable for v1.3; AMD `amd_energy` and CAP_SYS_RAWIO workarounds tracked for v1.4.
- **Snapshot-relay thread sequencing** — currently spawned in `lib.rs::run` *before* the Tauri builder. Must move into `.setup(...)` to capture the `AppHandle`. Verify the existing telemetry session-start emit at lines 171-175 still fires before the builder (it does — main thread, predates setup).
- **macOS template-icon mode** — colorful sparkline is on-brand but loses the auto dark/light adaptation a template image would give. Defer template-mode auto-detect to v1.4 polish.
- **Two-tray race during window-state plugin restore** — `tauri-plugin-window-state` restores window position before our setup closure runs; harmless because we call `hide()` after restore if `start_in_tray` is set. Verify in QA pass on each OS.
- **`chart_tab` round-trip downgrade** — a v1.3 user with `chart_tab = "disk"` downgrading to v1.2 silently resets to `cpu`. No data loss, no crash.
